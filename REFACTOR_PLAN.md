# WordPack 改造计划（2026-04）

## 背景与约束
- 当前截图翻译功能不可用，可直接删除后重做。
- 产品定位：企业内网使用，离线优先，流畅、通用、轻量。
- 当前技术栈：Python + pywebview + 前端 JS。
- 不做项：术语库、生词本替代、插件化接口平台。

## 总体目标
1. 修复并重建截图翻译为稳定可用链路。
2. 提升交互体验（Pot 风格截图：十字光标、拖拽选区、松开即识别）。
3. 降低体积与启动负担（主包轻量化、重 OCR 后置）。
4. 保持内网安全模型：翻译优先走公司内网接口，离线可兜底。

## 目标方案
- 双通道翻译：
  - 划词翻译：UIA 优先 + Clipboard 回退。
  - 截图翻译：全新链路（截图模块与 OCR 模块解耦）。
- OCR 单引擎固定：
  - 默认且唯一：Windows OCR（轻量、快速、系统能力）。

## 分阶段计划

### Phase 1：截图翻译重建（优先）
目标：把“不可用”变成“可用且顺滑”的基础版。

改造点：
1. 删除旧截图状态机相关死代码与问题路径。
2. 重建截图交互：
   - 开始截图后全屏透明覆盖层；
   - 鼠标变十字；
   - 左键拖拽选区；
   - 松开直接提交选区。
3. 重建最短闭环：
   - 选区裁剪图像 -> OCR -> 翻译 -> 主窗结果显示。
4. 提供明确错误反馈：
   - 取消、空选区、OCR失败、翻译失败分别提示。

交付标准：
- 截图翻译在 Win10/11 开发机可稳定使用。
- 80% 常见截图场景可完成识别与翻译。
- 无崩溃、无卡死、无主窗口丢失。

### Phase 2：OCR 统一到 Windows OCR
目标：简化链路，减少维护面与打包复杂度。

改造点：
1. 保留统一 OCR 接口，但仅启用 Windows OCR provider。
2. 移除 RapidOCR provider 与相关配置/依赖。
3. 配置项仅保留：
   - OCR 语言；
   - 超时参数。

交付标准：
- Windows OCR 识别稳定可用。
- 用户无需理解多引擎配置即可完成截图翻译。

### Phase 3：划词链路性能优化
目标：降低延迟抖动，提升取词成功率。

改造点：
1. 将 UIA 从 PowerShell 子进程改为进程内调用（Rust sidecar 优先）。
2. 保留 Clipboard 回退、剪贴板恢复与安全判断。
3. 优化触发策略（热键默认、图标可选）与防抖。

交付标准：
- 划词取词尾延迟明显下降。
- 成功率不低于当前版本。

### Phase 4：打包瘦身与发布准备
目标：主包轻量化。

改造点：
1. 移除 RapidOCR 相关依赖与扩展包流程。
2. 构建脚本统一为单包构建流程。

交付标准：
- 主包体积显著下降。
- 主包可独立运行。

## 回归测试策略

### A. 功能回归
1. 划词翻译：
   - 浏览器、Office、IDE、PDF 阅读器、终端。
   - 焦点控件与非焦点控件。
2. 截图翻译：
   - 明亮/暗色背景、小字体、中英混排、滚动页面。
   - 取消流程、Esc/右键取消、空选区。
3. 翻译链路：
   - 内网模型可用。
   - 离线模式降级行为正确。

### B. 兼容回归
1. Win10 / Win11。
2. 多分辨率、缩放比例（100/125/150%）。
3. 多显示器场景。

### C. 性能回归
1. 冷启动时间。
2. 截图到首条可见结果耗时。
3. 划词到结果耗时。
4. 内存占用峰值。

### D. 稳定性回归
1. 连续操作 200 次截图翻译无崩溃。
2. 连续切换模式、开关主窗口、托盘操作无卡死。
3. 快速重复热键触发无线程泄漏。

### E. 打包回归
1. 主包可运行并通过核心功能验证。
2. 升级安装不破坏配置与历史数据。

## 关键指标（验收）
1. 截图翻译可用率：>= 99%（常见场景）。
2. 截图链路首结果时间：较当前开发版显著下降。
3. 划词链路尾延迟：明显下降。
4. 主包体积：较当前方案显著缩小（Phase 4 达成）。
