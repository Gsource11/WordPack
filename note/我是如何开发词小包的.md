# 我是如何开发词小包的

词小包（WordPack）是我做的第一款桌面端应用。  
我的想法就是既能提升个人工作效率，又能体验AI智能体能做到什么程度。

整个过程我一直是同一个思路：我负责产品方向、交互细节、测试验收；AI 负责高速实现、重构和修复。  
我不是把开发外包给 AI，而是把 AI 变成我的工程搭子。  
所以这篇文章我就按真实过程讲：我是怎么从最早的粗糙版本，一路试错、换框架、重做 UI、磨细节，最后把词小包做出来的。

## 一、先做的不是“写代码”，而是把目标钉死

我给自己定的是四个能验收的目标，大概的想法：

1. 必须有两个核心场景：划词翻译、截图 OCR 翻译。  
2. 必须支持两条翻译能力：AI 翻译和离线模型翻译。  
3. 必须是低打扰体验：尽量不打断我当前操作。  
4. 必须能发布：最终要有安装包，不是本地演示脚本。

这一步看起来简单，但它决定了后面不会跑偏。  
我每次让 AI 改功能，都会盯着这四个目标问一句：这次改动到底是离目标更近，还是只是多了一个按钮。



## 二、前期：先追“能跑通”，再追“好体验”

前期进展很快。按我手记，当时十分钟就出了初版。  
但我马上发现：能跑和能用，差得非常远。

初版阶段最典型的问题有四个：

1. 长文本翻译不稳定。  
2. AI 翻译链路时好时坏。  
3. 离线模型导入报错，用户根本不知道哪一步错了。
4. UI简陋

我没有在初版上停太久，而是立刻进入“快速迭代+快速回归”循环。  
我那段时间几乎每天都在重复一个动作：提需求、让 AI 改、马上测、马上打回去再改。

我在手记里给这些迭代起了“二版、三版、四版……”的名字，本质不是为了记录版本号，而是逼自己保持节奏：  
每一轮只解决一组清晰问题，不把所有东西搅在一起。

### 例子：离线词典翻译

AI竟然自己生成一个离线词典。后来还是先自己去了解目前有哪些开源的离线词典翻译框架，找到一个Argos翻译框架，让AI合并到项目里。

## 三、中期：我把 UI 设计和代码落地彻底分开

我在 UI 这块踩过很多坑。  
最早我是“看到不爽就改代码”，结果是：今天改配色，明天改布局，后天又推翻。  
后来我把流程固定成两段：

1. 我应该先设计好UI，先在设计层定结构和风格。  
2. 再交给 Codex 只做落地实现。

利用 Pencil MCP，我当时不是“画个草图”，而是把主界面、设置页、历史页、气泡态都做成了可核对的目标。
与AI沟通，让它去画出我心中的UI。
每改一轮 UI，就看设计稿，直接暴露对齐问题、圆角断裂、层级冲突、暗色主题失真这些细节。

## 三、中后期：主动换技术路线

UI设计好后，落地时发现差距有点大，因为使用的Tkinter技术框架是python内置的基础 GUI 框架，界面风格较 "复古"，根本做不出来好看的。
了解原因后，就接着尝试新的技术框架去重构UI了，CustomTkinter方案，Flet 方案，也做过一整轮重构尝试，但实际体验和落地成本不理想。  
我没有硬扛，而是果断切到 pywebview 路线。

再UI重构能达到60分后，就决定使用pywebview路线了，后续就一直打磨UI页面。
改按钮功能合理，改布局合理性，改可用性等等。

## 五、后期：我把重心放在“体验手感”，不是“功能数量”

后期最难的已经不是翻译引擎，而是交互手感。  
词小包最核心的一串动作是：选中 -> 图标出现 -> 触发翻译 -> 气泡展示 -> 自动隐藏。  
这条链路里任何一个细节不稳，体感就会很差。

我在会话里连续提了很多“继续优化”，核心都在这里。  
我把它们整理成了几个稳定规则：

1. 图标出现前提是“有选区 + 鼠标静置在选中的文字上方”。  
2. 翻译中气泡和翻译结果气泡位置要一致。  
3. 支持点击触发和悬停触发两种方式。  
4. 鼠标进气泡时不自动隐藏，快速移离时要能快速隐藏。  
5. 同一选区要支持再次静置触发，不用先取消选中。  

这些规则看上去碎，但它们决定了“像工具”还是“像半成品”。

## 六、专门打磨了“看似小、其实非常重要”的窗口细节

词小包有一段时间会出现“窗口先在左上角闪一下，再出现到中间”。  
设置页偶尔也会在左边闪出一个边框。  
从功能上说，这不影响翻译；从产品感知上说，这很伤。

我在这类问题上花了不少轮次，因为工具一旦有“乱跳、闪烁、抖动”感，体验度就会快速下降。

这一阶段我做的不是加新功能，而是反复“抹毛刺”。  
我把这一步看得很重，因为这一步决定词小包能不能从“能用”走到“愿意一直开着用”。

## 七、收口阶段：我把项目从“可用”推进到“可维护”

我后期做了几件看起来不酷、但非常关键的事：

1. 统一品牌：中文名“词小包”，英文名“WordPack”。  
2. 统一入口：`app.py` 启动时先初始化日志和异常钩子。  
3. 统一发布：`scripts/build_release.ps1` 固定打包产物。  
4. 统一文档：`README.md` 明确运行、打包、模型导入路径。

## 八、AI 踩过的坑

### 坑 1：前期没有先做 UI 规划，导致后面被迫 UI 重构

我最开始只想着先把翻译功能做出来，UI 结构没有提前规划。  
结果就是前面看着进度很快，后面越改越乱，最后不得不进行一整轮 UI 重构。  
这件事让我吃了很大教训：桌面端工具哪怕先做功能，也要先把 UI 的骨架想清楚，不然返工成本很高。

### 坑 2：技术栈选择调研不够，很多决定都在“跟着 AI 走”

一开始 AI 检测到我本机 Python 环境，就直接按 Python 路线开干，我当时也默认接受了。  
后来我回头看，这一步我自己调研不够。  
比如从长期维护和桌面端能力看，我本来应该对 Rust、TypeScript 这些路线也认真评估一下，再决定。

### 坑 3：UI 重构方案当时主要听 AI 建议，我没有先看主流框架全景

我在 UI 重构阶段选了 pywebview，这个建议本身是 AI 给的。  
它确实帮我把当时的问题快速收住了，但我当时没有系统了解“现在主流桌面端 UI 框架有哪些、各自边界是什么”。  
这让我在后面做一些取舍时，心里没有一张完整对照表。  
这次之后我给自己定了一个规矩：AI 给方案可以，但最终技术路线我要自己先做一轮对比判断。

### 坑 4：一轮里塞太多目标，回归必炸

我试过一轮里同时改功能、改 UI、改打包。  
结果是出了问题时根本不知道哪一块引起的。  
后来我按会话分治：交互会话只做交互，UI 会话只做 UI，发布会话只做发布。

### 坑 5：忽略上下文长度，导致历史约束丢失

长会话到后面，AI 对早期约束记忆会变弱。  
多开会话，早期两三个会话都是用到爆，没有及时压缩上下文，导致会话完全不可用了。（疑似Codex没有自动压缩功能，后面到时有了，到90%就会压缩）
这个项目目前开了40多个会话。

### 坑 6：把“偶发”当“低优先级”

闪退、串写、剪贴板污染这类问题如果不追到底，后面一定反复爆。  
后来对“偶发高伤害”一律按 P0 处理，让AI一直跑。

## 九、总结

如果只看结果，词小包是一款翻译桌面应用：支持划词、截图、AI、离线、历史、设置、打包。  
但对我来说，真正的收获是我跑通验证了一套 与AI智能体 协作开发方法：

1. 我先定清楚目标和边界。  
2. 我把需求写成可执行、可测试的句子。  
3. 多轮小步快跑。  
4. 我把失败变成下一轮输入，而不是情绪。  
5. 我把“能用”继续推进到“可维护、可追溯、可发布”。

词小包不是一个一次性作品，而是我会继续迭代的长期工具。  
我已经验证了一件事：只要我把问题说清楚、把验收定清楚、把节奏控清楚，AI 就能把我的想法不断变成真正可用的产品。
